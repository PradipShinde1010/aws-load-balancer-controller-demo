Now lets do subdomain based routing

🚀 WHAT IS SUBDOMAIN ROUTING?
Instead of:
learndevops01.click/app1
learndevops01.click/app2

You use:
app1.learndevops01.click
app2.learndevops01.click
app3.learndevops01.click

🔍 PATH vs SUBDOMAIN (Quick Difference)
Feature					Path-Based	Subdomain-Based
URL					/app1		app1.domain.com
Isolation				Low		High
Production usage			Less		More
Microservices standard	❌			✅
App config issues		Common	Rare

👉 Key point:
Subdomains behave like separate apps, paths behave like one app with routes

This is because in the current code, 

from flask import Flask
app = Flask(__name__)

@app.route("/app1")
def home():
    return "Hello from APP-1 🚀"

@app.route("/")
def root():
    return "APP-1 root"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)

we have clearly written that, 
	if the path is /app1 ----> it will display ----> Hello from APP-1
	if the path is / ----> it will display ----> APP-1 root

So is why we are seeing that in the browser.

But in real-time we will not have something like this. We will write a common code which supports both, path based and sub-domain based routing.

Below is the updated code;
🔹 App1

from flask import Flask
app = Flask(__name__)

def response():
    return "Hello from APP-1 🚀"

# Root (subdomain case)
@app.route("/")
def root():
    return response()

# Path-based case
@app.route("/app1")
def app1():
    return response()

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)

🔹 App2

def response():
    return "Hello from APP-2 🔥"

@app.route("/")
def root():
    return response()

@app.route("/app2")
def app2():
    return response()

🔹 App3

def response():
    return "Hello from APP-3 ⚡"

@app.route("/")
def root():
    return response()

@app.route("/app3")
def app3():
    return response()
